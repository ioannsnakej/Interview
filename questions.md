  ### 1. Что такое load average? Что показывает эта метрика? Почему load average состоит из трёх значений?
  <details>
    <summary>Ответ</summary>

  ***LA (load average - средняя нагрузка системы)*** - параметр, определяющий среднее количество процессов, которые:
      
  - выполняются прямо сейчас (в состоянии running),
  - либо ожидают доступа к CPU или диску (в состоянии uninterruptible sleep).
      
  Значения показываются для последних 1, 5 и 15 минут. Это позволяет:

  - Оценить динамику нагрузки. Значения помогают понять, как процессы в системе нагружают CPU с течением времени.
  - Учесть не только активные процессы, но и те, которые ждут своей очереди. Это позволяет получить более полное представление о загруженности системы.
  - Учесть процессы, которые ожидают завершения операций ввода-вывода. Эти процессы могут значительно влиять на производительность системы, даже если они не занимают CPU напрямую. 
      
  #### Как посмотреть LA:
  - uptime
  - cat /proc/loadavg
  #### Как интерпритировать:
  Главное правило:
  **Сравнивайте load average с количеством ядер процессора.**
  
  Посмотреть количество ядер можно командой:
      
    nproc
  **Пример:**
  Если у вас N ядра, то:

  - load = N — система загружена на 100%
  - load < N — запас по ресурсам есть
  - load > N — ядрам не хватает времени, процессы начинают ждать
      
  #### Высокое значение не всегда значит, что процессор "зашит".
  На нагрузку влияют и другие факторы:

  |Причина|Описание|
  |-|-|
  |CPU-bound процессы|Программы, активно использующие процессор (компиляция, шифрование, вычисления).|
  |I/O wait | Задержки из-за медленного диска или файловой системы — процессы ждут завершения операций чтения/записи.|
  |Недостаток RAM | Когда система начинает использовать swap — растёт задержка и load.|
  |Блокировка ресурсов | Процессы ждут освобождения файлов, сокетов или блокировок БД.|
  |Много фоновых задач | Cron, systemd-службы, резервные копии, индексация и т.п.|
  
  Также влияние на расчета LA оказывает:
  1. Технология Hyper-Threading, которая делит одно физическое ядро на 2 логических
  2. Технология Turbo Bust, которая позволяет разгонять тактовую частоту процессора и работать на частоте выше заявленной, т.е. выше номинальной частоты (время на обработку одной задачи уменьшается).
  #### Как диагностировать:

  - htop покажет загруженность CPU и RAM, а также количество процессов в состоянии D (ожидание I/O)
  - iotop — отследит нагрузку на диск
  - iftop — покажет текущий сетевой трафик
  </details>

  ### 2. Что такое OOM killer? Для чего нужен? Как работает?
  <details>
    <summary>Ответ</summary>
    
  ***OOM Killer*** — механизм ядра Linux, который при исчерпании доступной памяти принудительно завершает отдельные процессы на сервере для освобождения RAM. Освободившаяся память передается ядру ОС, а после перенаправляется тому процессу, которому ее было недостаточно. Сообщение об этом отображается в /var/log/syslog (Debian/Ubuntu) или /var/log/messages (Centos/RHell).

  OOM Killer сканирует запущенные процессы и назначает каждому из них *oom_score* — показатель, который представляет его «вредность» или вероятность завершения. Некоторые факторы, на основе которых рассчитывается oom_score:

  - использование памяти (процессы, потребляющие больше памяти, получают более высокие оценки);
  - время работы процесса (более длительные процессы получают более низкие оценки);
  - привилегии root (процессы, запущенные под root, получают менее низкие оценки);
  - значение nice (процессы с более низким приоритетом (высокими значениями nice) получают более высокие оценки);
  - ручная настройка через oom_score_adj (настраивает вероятность завершения процесса).

  В первую очередь будут завершены недавно запущенные пользовательские процессы, которые требуют большого объема памяти и/или имеют множество дочерних процессов; в последнюю очередь будут завершаться системные процессы пользователя root, запущенные значительное время назад.
  В связи с тем, что OOM killer «убивает» процессы с помощью SIGKILL, что не дает возможности корректного завершения процессов, сохранения данных и пр., частое срабатывание этого механизма может приводить к серьезным последствиям в работе системе.

  #### Что делать:

  - Провести анализ работы сервера, баз данных, сайтов и тд с целью последующей оптимизации, чтобы устранить причину проблемы.
  - Если сервер уже оптимизирован, но памяти не хватает, то стоит задуматься о увеличении  объема RAM.
  - Также есть возможность установить приоритет определенному процессу, чтобы запретить OOM killer его завершать. Значение приоритета хранится в файле /proc/$PID/oom_adj. Для установки приоритета необходимо узнать PID конкретного процесса, после чего воспользоваться командой:
        
        echo -17 > /proc/PID_процесса/oom_adj

  Устанавливаемый приоритет в данном случае — -17. Более безопасный вариант — использовать приоритет -15. В этом случае процесс будет одним из последних, которые OOM killer завершит, но при этом отсутствует полный запрет на завершение, который в критических ситуациях может привести к kernel panic.
  </details>
  
  ### 3. Отличие hardlink от softlink (symlinks)
  <details>
    <summary>Ответ</summary>
    
  **Жёсткая ссылка (hardlink)** — это запись в каталоге, которая указывает непосредственно на инод файла. Она действует как отдельный файл и ссылается на конкретное место на жёстком диске. Это зеркальная копия оригинального файла.

  #### Как создать жесткую ссылку

      ln file hardlink
    
  Изменения, внесённые в hardlink, отображаются и в file, поскольку они ссылаются на один и тот же inode. Если удалить file, то содержимое файла будет доступно через hardlink

  #### Жесткие ссылки имеют следующее преимущества:

  - Эффективная компоновка: несколько жёстких ссылок могут совместно использовать один и тот же инод и блоки данных, что означает, что при создании нескольких жёстких ссылок на один файл не тратится лишнее место на диске.
  - Доступность файла: пока существует хотя бы одна жёсткая ссылка на файл, данные файла остаются доступными. Удалив одну жёсткую ссылку, мы не удаляем сам файл.
  - Производительность: жёсткие ссылки обычно срабатывают быстрее мягких, поскольку указывают непосредственно на индексный дескриптор и не требуют разрешения пути.
  - Прозрачность: жёсткие ссылки выглядят и функционируют как обычные файлы. Нет различия между исходным файлом и указывающими на него жёсткими ссылками.
  - Консистентность: все жёсткие ссылки на файл всегда актуальны. Любые изменения, внесённые через любую жёсткую ссылку, немедленно отражаются во всех остальных.

#### Некоторые ограничения:
  - Ограничения на каталоги: жёсткие ссылки можно создавать только для обычных файлов (не для каталогов или специальных файлов).
  - Ограничение на одну файловую систему: жесткие ссылки не могут распространяться на несколько файловых систем. Они работают только в том случае, если новая жесткая ссылка существует в той же файловой системе, что и исходная.

**Мягкая ссылка (softlink - символическая ссылка)** — это отдельный файл, содержащий путь к другому файлу или директории. В отличие от жёстких ссылок, мягкие ссылки не ссылаются напрямую на индексный дескриптор целевого файла, а имеют собственные индексные дескрипторы и содержат путь к целевому файлу или каталогу.

#### Как создать жесткую ссылку

      ln -s file softlink

> При создании символических ссылок следует указывать абсолютный путь, поскольку относительные пути не будут работать.

Изменения, внесённые через softlink, отражаются в file
    
#### Ссылки на каталоги

Поскольку связь мягкой ссылки является логической, а не дублированием, символические ссылки могут указывать на целые каталоги или на файлы на удалённых компьютерах. Жёсткие ссылки для этого не предназначены.

#### Символические ссылки имеют следующие преимущества:
  - Гибкость: мягкие ссылки могут указывать как на файлы, так и на директории, а также работать между разными файловыми системами или разделами.
  - Удобство: не составляет труда создавать ярлыки и организовывать файлы с помощью мягких ссылок. Они могут указывать на пока не существующие файлы и заработают, как только цель для них будет создана.
  - Низкие издержки: размер файлов символических ссылок небольшой, так как они содержат только путь к цели.
#### У символических ссылок есть и недостатки:
  - Хрупкость: они могут стать «висячими» ссылками, если цель будет перемещена, переименована или удалена.
  - Производительность: доступ к символическим ссылкам немного медленнее, чем к жёстким, поскольку требуется дополнительный шаг разрешения пути.
  - Небезопасность: если ими неправильно управлять, мягкие ссылки могут представлять угрозу безопасности, так как могут указывать на конфиденциальные файлы.
  - Разные индексные дескрипторы: мягкие ссылки имеют собственные айноды, из-за чего количество айнодов в системе может слишком увеличиться
  </details>

 ### 4. При создании нового файла система возвращает ошибку no space left device
  <details>
    <summary>Ответ</summary>
    
  **Ошибка «No space left on device»** при создании нового файла не всегда означает фактическое отсутствие места.
  "Ошибка no space left on device может возникать по трем причинам:
   1. закончилось место (проверяем df),
   2. закончились inode'ы (проверяем df -i),
   3. процесс держит удаленный файл (проверяем lsof +L1),
   4. достигнуты квоты пользователя (проверяем quota -v)".
  </details>
  
  ### 5. df сообщает о наличии 20 Гб занятого пространства, подсчёт занятого файлами места при помощи du даёт результат в 20 Мб. При каких обстоятельствах может возникнуть описанная ситуация?(deleted files)
  <details>
    <summary>Ответ</summary>
    
  При удаленном файле такое может быть. Процесс держит открытый файловый дескриптор на удаленный файл

  Ищем файл через

    lsof +L1

  При удалении файла, который в этот момент был «занят» процессом — его имя удаляется, но inode — остаётся в файловой системе до тех пор, пока не завершится процесс, который «держит» этот файл.

  Соответственно, что бы «освободить» уже удалённые файлы — необходимо перезапустить процесс, который этот файл держит.
  </details>

  ### 6. При создании нового файла пользователем система возвращает ошибку no space left on device
  <details>
    <summary>Ответ</summary>
  Достигнуты лимиты квот. Для каждого юзера и пользователя она своя может быть, жестко заданна. Нужно проверить квоты с помощью команд quota -v (для пользователя) и repquota -a (для администратора).
    
  **Квоты** — это механизм в Linux, который позволяет ограничивать, сколько дискового пространства может использовать пользователь или группа.

  - Без квот — любой пользователь может забить весь диск и положить систему.

  - С квотами — ты говоришь: "Вася, тебе можно максимум 5 ГБ". Когда Вася упирается в лимит, он больше не может писать, даже если на диске есть место.
  </details>

  ### 7. Что такое inode?
  <details>
    <summary>Ответ</summary>
    
  **Inode (индексный дескриптор)** — структура данных в файловой системе Linux, в которой хранятся все метаданные файла, кроме его имени и содержимого.
    
  Что хранит inode:
    
  - права доступа (rwx)
  - владелец (UID) и группа (GID)
  - размер файла
  - временные метки: atime (доступ), mtime (изменение), ctime (статус)
  - тип файла: обычный, директория, symlink, устройство и т.д.
  - количество жёстких ссылок
  - указатели на блоки данных на диске

  Имя файла хранится в структуре каталога как пара "имя → номер inode". Сам inode имени не знает. Номера inode уникальны в пределах раздела. Если закончились свободные inode, создать новый файл нельзя, даже если место на диске есть.

    ls -li                # показать inode файлов
    df -i                 # показать использование inode по разделам
  </details>
  
  ### 8. Что такое файловые дескрипторы? Как посмотреть, сколько их использует процесс? Что делать, если возникает ошибка "too many open files"?
  <details>
    <summary>Ответ</summary>
  
**Файловый дескриптор** - неотрицательное целое число, которое используется в интерфейсе между пространством пользователя и пространством ядра (kernel) для идентификации ресурсов файла / сокета. Когда создаётся новый поток ввода-вывода, ядро возвращает процессу, создавшему поток ввода-вывода, его файловый дескриптор. Важно отметить, что файловые дескрипторы не ограничиваются только файлами. Они также могут ссылаться на каталоги, сокеты, каналы (pipes), устройства ввода-вывода и даже некоторые специфические для процесса ресурсы, такие как файлы процесса и области памяти.

Иными словами **Файловый дескриптор** — это номер в таблице открытых файлов, который ядро даёт процессу, чтобы тот мог обращаться к файлу, сокету или каналу, не указывая каждый раз путь.

Операционная система автоматически создает три файловых дескриптора для каждого процесса: 

- `0` — стандартный ввод (stdin);
- `1` — стандартный вывод (stdout);
- `2` — стандартный поток ошибок (stderr).

Как посмотреть?:

- Для общего количества — `cat /proc/sys/fs/file-nr`. В файле три числа: текущие использованные дескрипторы, выделенные дескрипторы и максимальное значение.
- Для конкретного процесса — `lsof -p $PID | wc -l`. В выводе — количество открытых дескрипторов для указанного процесса.
- Для текущего процесса — `ls -l /proc/$$/fd`

Ошибка `too many open files` возникает, когда процесс исчерпал лимит на количество открытых файловых дескрипторов.  
Это может быть как лимит самого процесса, так и пользователя, systemd-сервиса или общесистемный лимит.  

- Сначала проверить лимиты для данного процесса:

      grep "Max open files" /proc/PID/limits
      # Показывает, сколько файлов данному процессу разрешено открыть.
      # Две колонки: soft limit (текущий) и hard limit (максимум, до которого можно поднять без прав root).
- Посмотреть лимиты для текущего пользователя (для всех его процессов):

      ulimit -n
      # Лимит на количество открытых файлов для текущей сессии. Если процесс запущен из этой сессии — унаследует это значение.
- Посмотреть лимиты systemd.

      grep LimitNOFILE /etc/systemd/system/<service_name>.service
      # Systemd может жёстко ограничивать сервис через LimitNOFILE.
      # Если там стоит маленькое число — процесс упрётся, даже если в системе всё свободно.
- Посмотреть общесистемные лимиты.

      cat /etc/security/limits.conf | grep nofile
      #  Задаёт ограничения на уровне пользователей или групп.
      # Формат: пользователь тип ресурс значение
- Посмотреть системный максимум и сколько открыто

      cat /proc/sys/fs/file-max
      # Максимальное количество открытых файлов для всей системы в целом.
      # Если упёрлись сюда — проблема глобальная.
      cat /proc/sys/fs/file-nr   # сколько открыто сейчас
- Посмотреть, сколько открыто у конкретного процесса

      lsof -p PID | wc -l
      # Количество открытых файловых дескрипторов у процесса.
      # Если близко к лимиту из пункта 1 — процесс скоро упадёт с too many open files.

Лечение:

- Поднять лимиты для текущей сессии (временно)

      ulimit -n 65535
      # Поднимает лимит только для текущего терминала и его дочерних процессов.
      # После перезахода сбросится.
- Поднять лимиты постоянно для пользователя

      echo "ivan soft nofile 65535" >> /etc/security/limits.conf
      echo "ivan hard nofile 65535" >> /etc/security/limits.conf
      # Работает при входе в систему.
      # Требует перелогина.
- Поднять лимиты для systemd-сервиса

      # В файле /etc/systemd/system/имя_сервиса.service добавить:
      [Service]
      LimitNOFILE=65535
      # Затем:
      systemctl daemon-reload
      systemctl restart имя_сервиса
      # Systemd переопределяет limits.conf для своих сервисов.
      # Без этой настройки сервис будет жить по своим правилам.
- Поднять лимиты глобально для всей системы

      echo "fs.file-max = 2097152" >> /etc/sysctl.conf
      sysctl -p
      # Поднимает потолок для всего ядра.
      # Нужно, если упёрлись в file-max.
  </details>
  
  ### 9. Что такое процесс-зомби и процесс сирота?
  <details>
    <summary>Ответ</summary>
    
  **Процесс-зомби (Zombie, defunct process)** - это процесс, который завершил свое выполнение, но его запись еще остается в таблице процессов, это происходит потому, что родительский процесс еще не прочитал статус завершения дочернего процесса с помощью системного вызова wait() или его вариантов.  
  Характеристики зомби:

  - Занимает только запись в таблице процессов (PID, статус завершения, учетная информация)
  - Не занимает оперативную память (ресурсы уже освобождены)
  - Не может быть завершен сигналом (уже мертв)
  - В выводе ps помечен как `defunct` или `z`
  - Зомби можно сделать самостоятельно. 

 Как бороться с зомби?  

1. Разработка:  
- Использовать wait()/waitpid() в родительском процессе
- Устанавливать обработчик сигнала SIGCHLD
- Использовать двойной fork() для демонов

2. Администрирование:  
- Перезапустить родительский прцоесс
- Перезагрузка (если зомби занимает все PID)
  
 **Процесс-сирота (Orphan process)** - это процесс, родительский процесс которого завершился раньше него, в Unix/Linux такие процессы автоматически "усыновляются" процессом init(PID 1) или systemd(в современных системах).

| Зомби | Сирота |
|-|-|
|Родитель жив, но не вызвал wait()|Родитель мертв (усыновлен init/systemd)|
|Не занимает память|Занимает память|
|Не использует CPU|Использует CPU|
|Опасность - временное потребление PID|Опасность - нет (авто усыновление)|
|Решение - Родитель должен вызвать wait()|Автоматически решается ОС|

 - Кратковременные зомби - нормальное явление, они существуют миллисекунды
 - Долгоживущие зомби - проблема программы (баг), когда родитель не вызывает wait()
 - Сироты часто используются демонами для определения от терминала.
  </details>

### 12. на сервере с 16 ГБ RAM начало активно расти потребление памяти, и система начала активно использовать swap (файл подкачки). Означает ли это, что OOM Killer вот-вот сработает? Или есть ещё запас? Почему?   
<details>
  <summary>Ответ</summary>

  Нет, появление swap не означает, что OOM Killer вот-вот сработает. OOM Killer — это механизм крайнего случая, когда кончилась и RAM, и swap.  
  1. Сначала при нехватке RAM ядро начинает вытеснять редко используемые страницы памяти в swap. Это нормальный рабочий процесс. Система может жить с активным swap месяцами.
  2. Потом, если и swap начал заполняться, ядро пытается ужимать кэши и буферы.
  3. И только когда совсем всё кончилось (RAM под завязку, swap под завязку, и освободить нечего), ядро в панике вызывает OOM Killer, чтобы кого-нибудь убить и освободить память.
</details>

### 13. Что такое systemd?   
<details>
  <summary>Ответ</summary>

  **Systemd** — это системный менеджер (система инициализации) в современных Linux-дистрибутивах. Его задачи:

  1. Запускает систему — самый первый процесс (PID 1), который стартует после загрузки ядра.
  2. Управляет сервисами — запускает, останавливает, перезапускает, следит за ними.
  3. Параллельный запуск — в отличие от старого SysV Init (который запускал всё по очереди), systemd запускает сервисы параллельно, поэтому система грузится быстрее.
  4. Логирование — через journald собирает все логи в одном месте.
  5. Управление зависимостями — может сказать: "Пока сеть не поднялась, nginx не запускай".

  Как проверить, что в системе используется `systemd`?

    ps -p 1
</details>

### 14. Как написать свой unit-file?
<details>
  <summary>Ответ</summary>
  
  **Unit-файлы** — это конфиги, которые описывают, как systemd должен управлять твоим сервисом. Лежат они в:

  - /etc/systemd/system/ — для пользовательских сервисов (ты сюда кладёшь свои)
  - /lib/systemd/system/ — для системных (от пакетов)

#### [Unit] — мета-информация  
- `Description` — что это за сервис (увидишь в systemctl status)
- `After=network.target` — запускать только после того, как поднялась сеть

#### [Service] — как запускать  
- `Type=simple` — процесс просто запускается и работает в фоне (самый частый тип)
- `User/Group` — от какого пользователя запускать (важно для безопасности!)
- `WorkingDirectory` — рабочая папка (относительные пути будут от неё)
- `ExecStart` — полная команда запуска
- `Restart=always` — перезапускать, если упадёт
- `RestartSec=10` — подождать 10 секунд перед перезапуском

#### [Install] — когда запускать  
- `WantedBy=multi-user.target` — запускать при загрузке в многопользовательском режиме (обычный запуск системы)

#### Коротко для запоминания
1. **Systemd** — это главный процесс (PID 1), который управляет всем.
2. **Unit-файл** — текстовый конфиг с тремя секциями:  

- [Unit] — описание и зависимости
- [Service] — как запускать (команда, юзер, рестарты)
- [Install] — когда запускать (автозагрузка)

3. Главные команды: `start`, `stop`, `restart`, `enable`, `status`, `journalctl -u`.
</details>

### 15. Отличия systemd и init?
<details>
  <summary>Ответ</summary>

Init — старый способ (был в Ubuntu до 2015 года, в CentOS до 2014).  
Systemd — новый, современный способ (везде сейчас).  
Коротко: Init запускал всё последовательно и медленно, systemd запускает параллельно и быстро  

|Характеристика|	|Init (SysV)|	Systemd|
|-|-|-|
|Скорость загрузки	|Медленная (всё по очереди)|	Быстрая (параллельно)|
|Тип файлов	|Скрипты (bash)|	Конфиги (текстовые .service файлы)|
|Зависимости	|Нужно самому прописывать номера (S01, S02)|	Сам определяет через After=, Requires=|
|Логи|	В разных файлах (/var/log/syslog, /var/log/messages)|	Централизованно через journald (одна команда journalctl)|
|Управление|	service nginx start	|systemctl start nginx|
|Мониторинг|Нет встроенного|	Сам следит за процессами, перезапускает если упали|
|На каком языке|	Bash-скрипты	|C (быстрый, но сложнее)|

### На примере: запуск nginx  
#### Init (старый способ)

1. Лежит скрипт /etc/init.d/nginx (это bash-скрипт).
2. Чтобы добавить в автозагрузку, нужно было создавать симлинки с номерами:

        ln -s /etc/init.d/nginx /etc/rc3.d/S20nginx
3. Цифра 20 означала "порядок запуска". Всё руками.
4. Запуск:

        service nginx start
        # или
        /etc/init.d/nginx start
#### Systemd (новый способ)  

1. Лежит файл /etc/systemd/system/nginx.service (простой текстовый конфиг).
2. Там написано:

        ini
        [Unit]
        Description=nginx
        After=network.target
        
        [Service]
        ExecStart=/usr/sbin/nginx
        Restart=always
3. Запуск:

        systemctl start nginx

  ### 1. Что такое IP-адрес, маска подсети, шлюз по умолчанию? 
  <details>
    <summary>Ответ</summary>
      
  **IP-адрес (IP от англ. Internet Protocol)** — цифровой идентификатор, присваиваемый устройству, которое работает в условиях публичной или локальной сети на основе стека протоколов `TCP/IP`.   
  
  **Маска подсети (subnet mask)** — параметр, который определяет, какая часть IP-адреса относится к сети, а какая — к хосту.  
  Пример: IP-адрес — 192.168.1.10, маска подсети — 255.255.255.0. Маска указывает, что первые три октета (192.168.1) — это адрес сети, а последний (10) — идентификатор устройства.  
  Маска подсети необходима для правильного разделения сети, чтобы устройства могли обмениваться данными с другими устройствами в той же подсети, а для устройств в других подсетях данные передавались через маршрутизаторы.  

  **Шлюз по умолчанию (англ. Default gateway)** — узел компьютерной сети, на который пакет отправляется, если маршрут к сети назначения пакета неизвестен (не задан явным образом в таблице маршрутизации хоста или не соответствует IP-адресу назначения пакета).  
  В сетях TCP/IP роль шлюза, как правило, выполняет маршрутизатор, соединяющий одну сеть с другой для их взаимодействия.  
  Пример работы: когда компьютер пытается подключиться к ресурсу в интернете, но его адрес не соответствует ни одной локальной подсети, данные отправляются через шлюз.   
  </details>

  ### 2. Сколько адресов в следующих подсетях?  
    192.168.5.0/24
    192.168.5.0/23
    192.168.5.0/19
  <details>
    <summary>Ответ</summary>
  
  `2^(32-N)-2`, где

  - N маска: `/24`, `/23`, `/19`;
  - 32 бит в маске;
  - `-2` зарезервированных адреса: 1 адрес сети и 1 broadcast.
Следовательно:

|сеть|математика|ответ|
|-|-|-|
|192.168.5.0/24|2^(32-24)-2|= 254 хостов|
|192.168.5.0/23|2^(32-23)-2|= 510 хостов|
|192.168.5.0/19|2^(32-19)-2|= 8190 хостов|
  </details>
  
  ### 3. Что такое 127.0.0.1 адрес? Для чего нужен? 
  <details>
    <summary>Ответ</summary>
      
  `127.0.0.1` адрес или `localhost` доменное имя, а также зарезервированная сеть 127.0.0.1/8 частных IP адресов предназначены для тестирования программы на той же физической машине, где она запускается.  
  Использование адреса 127.0.0.1 позволяет устанавливать соединение и передавать информацию для программ-серверов, работающих на том же компьютере, что и программа-клиент, независимо от конфигурации аппаратных сетевых средств компьютера (не требуется сетевая карта, модем, и прочее коммуникационное оборудование, интерфейс реализуется при помощи драйвера псевдоустройства в ядре операционной системы)

Так же адрес 127.0.0.1 устанавливается для запрета доступа к сервису из внешней сети. Например:

    docker run -d -p 127.0.0.1:3306:3306 mysql
  </details>

   ### 4. Чем отличается TCP от UDP? Что лучше?
  <details>
    <summary>Ответ</summary>
      
  **TCP** – транспортный протокол передачи данных в сетях TCP/IP, предназначен для управления передачей данных интернета. Пакеты в TCP называются сегментами. Ориентирован на соединение, используется для передачи данных (электронная почта, файлы, сообщения). При определении потери пакетов будет выполнен перезапрос потерянных пакетов.

  **UDP** – транспортный протокол, передающий сообщения-датаграммы без необходимости установки соединения в IP-сети. Не ориентирован на установление соединения, используется в потоковой передаче данных (IPTV, VoIP). При потере пакетов перезапроса потерянных пакетов не происходит.

Нельзя сказать, что `TCP` лучше `UDP`, т.к. данные транспортные протоколы используются для различных типов передачи трафика.
</details>

  ### 5. Как работает DNS, зачем он нужен?
  <details>
    <summary>Ответ</summary>
      
 **DNS (Domain Name System)** — система доменных имён, которая преобразует человекопонятные доменные имена (например, example.com) в IP-адреса, понятные сетевым устройствам (например, 192.0.2.1). Без DNS пользователям пришлось бы запоминать и вводить цифровые IP-адреса для каждого сайта или сервера, что неудобно.
</details>

  ### 6. Почему DNS использует UDP??
  <details>
    <summary>Ответ</summary>
  Пока устанавливается TCP-сессию, можно 3 раза отправить UDP пакет и получить его обратно.  
    
  Все реализации DNS серверов должны поддерживать использование обоих протоколов транспортного уровня (TCP и UDP). Большинство DNS-запросов будет обрабатываться с использованием протокола UDP, исключение составляют трансфер зоны (Query type AXFR) и ответы сервера, превышающие 512 байт на одно сообщение. На вопрос "зачем?" ответ простой -- чтобы не использовались для DDoS.
    </details>
    
  ### 7. Почему корневых серверов DNS 13?
  <details>
    <summary>Ответ</summary>
  Число 13 — историческое ограничение из ранних спецификаций DNS (RFC 1035), когда ответ должен был помещаться в 512 байт UDP-пакета. Инженеры посчитали, что максимально можно разместить 13 записей о серверах. Сегодня это число осталось как дань традиции, но физически за 13 логическими именами стоят тысячи серверов по всему миру, объединенных технологией Anycast. Это обеспечивает высокую отказоустойчивость и низкую задержку для пользователей из любой точки планеты.
    </details>
    
  ### 8. Расскажите модель OSI. Какие уровни знаете? С чем работает DevOps на каждом уровне?
  <details>
    <summary>Ответ</summary>

  **Модель OSI** - Эталонная модель взаимодействия открытых систем, состоящая из 7 уровней. Снизу вверх (от физики к приложению).  
  Уровни:  

 1. Physical (Физический) - провода  
    Что делает: Передаёт биты (0 и 1) по проводам, радиочастотам, оптике. Напряжение, разъёмы, типы кабелей.  
    С чем работает DevOps:
    - Даунтайм из-за перебитого кабеля в ЦОДе
    - Проблемы с оптическими трансиверами (SFP)
    - Проверка линка (ethtool, mii-tool)
      
    Пример: Пропал пинг до сервера, едешь в дата-центр перетыкать патч-корд.

 2. Data Link (Канальный) - MAC-адреса  
    Что делает: Передаёт кадры (frames) между двумя соседними устройствами в одной сети. Работает с MAC-адресами.  
    Протоколы: Ethernet, Wi-Fi (IEEE 802.11), ARP (частично).  
    С чем работает DevOps:
    - Коммутаторы (L2-свитчи)
    - VLAN'ы (тегирование 802.1Q)
    - MAC-адреса (спуфинг, фильтрация)
    - Диагностика: `arp -a`, `ip neigh`

    Пример: Настроил VLAN для разделения трафика между окружениями (dev/stage/prod).
    
 3. Network (Сетевой) - IP-адреса  
    Что делает: Маршрутизация пакетов между разными сетями. Работает с IP-адресами.  
    Протоколы: IP (IPv4, IPv6), ICMP (ping), OSPF, BGP.  
    С чем работает DevOps:  
    - IP-адресация, маски подсетей
    - Маршрутизаторы (L3)
    - NAT (выход в интернет из приватных подсетей)
    - Балансировщики (L3/L4)
    - Диагностика: `ping`, `traceroute`, `ip route`, `netstat -rn`

    Пример: Настроил Ingress-контроллер в Kubernetes, который маршрутизирует трафик к сервисам.
    
 4. Transport (Транспортный) - порты  
    Что делает: Обеспечивает доставку данных между приложениями. Контроль портов, сегментация, надёжность.  
    Протоколы: TCP (надёжный, с подтверждением), UDP (быстрый, без подтверждения).  
    С чем работает DevOps:  
    - Порты TCP/UDP (80, 443, 3306, 5432)
    - Состояния соединений (LISTEN, ESTABLISHED, TIME_WAIT)
    - Балансировщики L4 (NLB, HAProxy в TCP-режиме)
    - Network Policies в Kubernetes (разрешить трафик только на порт 8080)
    - Диагностика: `ss -tulpn`, `netstat`, `telnet`, `nc`

    Пример: Настроил фаервол, который разрешает входящие соединения только на 443 порт (HTTPS).
    
 5. Session (Сеансовый) - сессии  
    Что делает: Управление сеансом связи: установка, поддержка, завершение. Синхронизация.  
    Протоколы: NetBIOS, RPC, SIP (частично).  
    С чем работает DevOps:  
    - TLS/SSL-сессии (рукопожатие, время жизни сессии)
    - HTTP-сессии (куки, JWT)
    - Балансировщики L7 (session stickiness/persistence)

    Пример: Настроил sticky sessions на балансировщике, чтобы пользователь не терял корзину при переключении между бэкендами.
    
 6. Presentation (Представления) - шифрование/кодировка  
    Что делает: Преобразование данных в понятный для приложения формат. Шифрование, сжатие, кодировка.  
    Протоколы: SSL/TLS, JPEG, GIF, ASCII, UTF-8.  
    С чем работает DevOps:  
    - SSL-сертификаты (Let's Encrypt, mTLS)
    - Форматы данных: JSON, YAML, XML, Protobuf
    - Сжатие (gzip, brotli в веб-серверах)
    - Кодировки (UTF-8 в базах данных)

    Пример: Выпустил сертификат для сайта, настроил автоматическое продление через cert-manager в Kubernetes.
    
 7. Application (Прикладной) - HTTP, DNS  
    Что делает: Взаимодействие с пользователем и приложениями. Самый близкий к пользователю уровень.  
    Протоколы: HTTP/HTTPS, DNS, FTP, SMTP, SSH.  
    С чем работает DevOps:  
    - Веб-серверы (Nginx, Apache)
    - DNS-записи (A, CNAME, MX)
    - SSH-доступ к серверам
    - API-шлюзы (Kong, Ambassador)
    - Ingress-контроллеры в Kubernetes
    - Балансировщики L7 (ALB, HAProxy в HTTP-режиме)

    Пример: Настроил Ingress для микросервиса, прописал DNS, проверил, что сайт открывается по HTTPS.

Вопрос: На каком уровне работает ping?  
Ответ: ICMP — это протокол сетевого уровня (L3) , но часто для диагностики используют утилиту ping, которая работает на прикладном уровне.

Вопрос: А DNS?  
Ответ: DNS работает на прикладном уровне (L7) , хотя использует порт 53 и UDP/TCP (L4).

Вопрос: Где в этой модели живёт Kubernetes?  
Ответ: K8s работает на всех уровнях от 3 до 7:

- L3: Network Policies, Service (ClusterIP)
- L4: NodePort, LoadBalancer
- L7: Ingress, Gateway API
  </details>
  
  ### 9. Что происходит, когда вводишь yandex.ru в браузере??
  <details>
    <summary>Ответ</summary>
Сначала браузер ищет IP, потом устанавливает соединение, потом шифрует, потом отправляет запрос

#### Этап 1: Разбор URL и протокол
Браузер смотрит на адресную строку. Структура: <протокол>://<хост>/<путь>. Например: https://yandex.ru/pogoda/samara.  
Если протокол не указан (написали просто yandex.ru), браузер проверяет список HSTS — это список сайтов, которые должны открываться только по HTTPS. Яндекс там есть, поэтому сразу будет HTTPS.  

#### Этап 2: Поиск IP-адреса (DNS)
Браузеру нужен IP-адрес сервера, чтобы установить соединение. Доменное имя yandex.ru нужно преобразовать в IP.  
Поиск идёт по цепочке (от себя в интернет):  
- Кэш браузера — может, недавно уже заходили.
- Файл hosts на компьютере (C:\Windows\System32\drivers\etc\hosts или /etc/hosts в Linux).
- Кэш операционной системы.
- Кэш роутера (если есть).
- DNS-сервер интернет-провайдера.

Если нигде нет, провайдер начинает рекурсивный запрос:  
Провайдер → Корневой DNS-сервер (точка .). Корневой говорит: "Я не знаю, спроси у сервера, который отвечает за зону .ru".  
Провайдер → DNS-сервер зоны .ru. Тот говорит: "Я не знаю, спроси у сервера, который отвечает за домен yandex.ru".  
Провайдер → NS-сервер Яндекса. Тот знает и отдаёт IP-адрес (например, 5.255.255.70).

IP-адрес возвращается обратно по цепочке и сохраняется в кэше на всех уровнях.

#### Этап 3: Формирование запроса (Уровни OSI)
Теперь у нас есть IP, нужно отправить запрос. Здесь в дело вступает сетевая модель. 

|Уровень OSI|	Что происходит|	Пример|
|-|-|-|
|Прикладной (L7)|	Браузер формирует HTTP-запрос.|	GET /pogoda/samara HTTP/2, заголовок HOST: yandex.ru|
|Транспортный (L4)|	Добавляет порт (443 для HTTPS).| Выбирает протокол TCP (надёжный).	Исходный порт: случайный (34567), целевой порт: 443|
|Сетевой (L3)|	Добавляет IP-адреса: свой (192.168.1.10) и сервера (5.255.255.70).|	Пакет знает, куда лететь|
|Канальный (L2)|	Добавляет MAC-адреса. Если сервер не в локальной сети, пакет уходит на шлюз (роутер).|	Роутер смотрит свою таблицу маршрутизации и отправляет дальше|

Иными словами:
1. L7 (Приложение): Браузер формирует "хотелку" — GET /pogoda/samara.
2. L4 (Транспорт): Браузер говорит системе: "Упакуй эту хотелку и отправь её на порт 443 (комната HTTPS)".
3. L3 (Сеть): Система смотрит: "Кому слать? IP Яндекса: 5.255.255.70, мой IP: 192.168.1.10. Поехали".
4. L2 (Канал): Компьютер смотрит: "А где этот 5.255.255.70? В моей локальной сети его нет. Значит, отдаю всю пачку главному курьеру — шлюзу (роутеру, 192.168.1.1)". Роутер дальше сам разбирается, кидает пакет провайдеру, тот — следующему узлу, и так по цепочке.

#### Этап 4: TLS-рукопожатие (установка безопасного соединения)
Так как у нас HTTPS, перед отправкой данных нужно договориться о шифровании.  

1. Client Hello — браузер говорит серверу: "Привет, давай шифровать. Я умею такие-то версии TLS и алгоритмы".
2. Server Hello — сервер отвечает: "Ок, давай используем TLS 1.3, вот мой публичный ключ и сертификат (подписанный центром сертификации)".
3. Проверка сертификата — браузер проверяет, что сертификат подлинный (не просрочен, выпущен для yandex.ru, подписан доверенным центром).
4. Обмен ключами — браузер генерирует общий секретный ключ, шифрует его публичным ключом сервера и отправляет.
5. Финиш — обе стороны подтверждают, что всё ок. Соединение установлено, данные теперь шифруются.

#### Этап 5: Отправка HTTP-запроса и получение ответа
По защищённому каналу браузер отправляет тот самый HTTP-запрос:

    text
    GET /pogoda/samara HTTP/2
    HOST: yandex.ru
    User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) ...
    Accept: text/html,application/xhtml+xml...
Сервер Яндекса принимает запрос, смотрит заголовок HOST, понимает, что это yandex.ru, и отдаёт нужную страницу (или отдаёт бэкенду, который её сгенерирует).  
Ответ приходит обратно (с HTML-кодом страницы), браузер его отрисовывает — и ты видишь погоду в Самаре.  

#### Шпаргалка для собеседования (самое главное)

1. DNS-запрос — браузер ищет IP (кеш → hosts → провайдер → корневые DNS).
2. TCP-соединение — открывается сокет по IP:порт.
3. TLS — если HTTPS, обмен сертификатами и ключами.
4. HTTP-запрос — браузер просит конкретную страницу (GET /pogoda/samara).
5. Ответ — сервер отдаёт HTML.

Почему это важно для DevOps? Когда сайт падает, нужно понимать, на каком этапе проблема:

- DNS не резолвится — провайдер или кеш.
- Не коннектится по порту — фаервол или сервер не слушает.
- TLS-ошибка — сертификат протух.
- 404/500 — проблемы с приложением или бэкендом.
  
  </details>

  ### 10. Что такое HTTP/HTTPS? В чем разница?
  <details>
    <summary>Ответ</summary>
  
    **HTTP** — это протокол передачи данных, по которому браузер общается с сервером. Когда ты открываешь сайт, браузер отправляет HTTP-запрос, сервер отвечает HTTP-ответом (обычно с HTML-страницей).  
    **HTTPS** — это тот же HTTP, но поверх SSL/TLS-шифрования. То есть данные передаются в зашифрованном виде.

    |Критерий|	HTTP|	HTTPS|
    |-|-|-|
    |Шифрование|	Нет (plain text)|	Есть (TLS/SSL)|
    |Порт по умолчанию|	80|	443|
    |Сертификат|	Не нужен	|Нужен SSL-сертификат|
    |Безопасность|	Низкая (можно прослушать)|	Высокая (данные защищены)|
    |Скорость|	Выше (нет шифрования)|	Чуть ниже (рукопожатие + шифрование)|
    |SEO|	"Небезопасный сайт" в браузере|	"Замок" в адресной строке, лучше ранжируется|

  ### 11. Как устроен HTTPS-запрос?
   <details>
    <summary>Ответ</summary>
    
Весь HTTPS — это HTTP внутри TLS. Сначала устанавливается защищённое соединение, потом внутри него летит обычный HTTP-запрос.  

#### Шаг 1: TCP-рукопожатие  
Как обычно, клиент и сервер договариваются о TCP-соединении (SYN, SYN-ACK, ACK) — это L4.

#### Шаг 2: TLS-рукопожатие (самое важное)  
1. Client Hello  
Браузер говорит серверу: "Привет, я хочу защищённое соединение. Поддерживаю версии TLS 1.2, 1.3, вот список шифров, которые знаю".
2. Server Hello  
Сервер отвечает: "Ок, давай использовать TLS 1.3, вот выбранный шифр. А вот мой SSL-сертификат и публичный ключ".
3. Проверка сертификата
Браузер смотрит сертификат:
- Не просрочен?
- Выдан именно для этого домена (yandex.ru)?
- Подписан доверенным центром сертификации (CA)?  
  Если всё ок — серверу можно доверять.
4. Обмен ключами  
Браузер генерирует секретный ключ для текущей сессии, шифрует его публичным ключом сервера и отправляет. Только сервер сможет расшифровать своим приватным ключом.
5. Финиш  
Оба подтверждают: "Соединение защищено, дальше всё шифруем этим сессионным ключом".  
#### Шаг 3: HTTP-запрос внутри шифрованного туннеля  
Теперь, когда защищённый канал установлен, браузер отправляет обычный HTTP-запрос (GET /, заголовки, куки), но он уже зашифрован.  
#### Шаг 4: Ответ сервера  
Сервер обрабатывает запрос и отправляет ответ (HTML, JSON, картинку) — тоже в зашифрованном виде.  
#### Шаг 5: Браузер расшифровывает и показывает страницу

### 12. Что такое SSL-сертификат?
   <details>
    <summary>Ответ</summary>

 **SSL-сертификат** - это цифровой паспорт сайта. Он подтверждает, что сервер — это действительно нужный сайт, а не мошенник. В нём указано:
  - Кому выдан (домен)
  - Кем выдан (центр сертификации, например Let's Encrypt)
  - Срок действия
  - Публичный ключ (для шифрования)

Браузеры хранят список доверенных центров сертификации. Если сертификат подписан кем-то из этого списка — браузер доверяет.

### 13. Что такое TLS?
   <details>
    <summary>Ответ</summary>

   **TLS (Transport Layer Security)** — это протокол, который обеспечивает шифрование, аутентификацию и целостность данных между двумя приложениями (чаще всего между браузером и сервером).  
  Простыми словами: TLS — это магия, которая превращает обычный HTTP в HTTPS.

  |Без TLS|	С TLS|
  |-|-|
  |Данные передаются как есть (открытый текст)|	Данные шифруются|
  |Любой может перехватить и прочитать|	Только получатель может расшифровать|
  |Ты не знаешь, кому отправляешь данные	|Сертификат подтверждает, что сервер настоящий|
  |Данные могут подменить в пути|	Если данные подменили — клиент это заметит|

  TLS — это криптографический протокол, который обеспечивает безопасную передачу данных по сети. Он работает поверх TCP, перед HTTP. Использует асимметричное шифрование для обмена ключами и симметричное для самих данных. Благодаря TLS мы получаем HTTPS, где данные защищены от перехвата и подмены, а сертификаты подтверждают подлинность сервера.
  </details>

  ### 14. Что такое NAT?
   <details>
    <summary>Ответ</summary>

   **NAT (Network Address Translation — преобразование сетевых адресов)** — это механизм, который позволяет нескольким устройствам в локальной сети выходить в интернет, используя **один общий публичный IP-адрес**.  
    Он подменяет внутренние IP-адреса на внешний и ведёт таблицу соответствий, чтобы возвращать ответы обратно нужному устройству. Это решает проблему нехватки IPv4-адресов и даёт базовую безопасность, скрывая внутреннюю структуру сети.

   #### Зачем он нужен?  
  1. Экономия IPv4-адресов  
  В мире осталось очень мало свободных IPv4-адресов (это те самые адреса вида 93.184.216.34). Если бы каждому устройству в мире (телефону, ноутбуку, холодильнику) давали свой уникальный публичный IP — их бы не хватило.  
NAT позволяет тысячам устройств сидеть за одним публичным IP. Провайдер даёт тебе один IP на квартиру, а роутер сам раздаёт внутренние адреса всем твоим устройствам.  
  2. Безопасность (скрытие внутренней сети)  
  Из интернета не видно, что у тебя внутри: сколько компьютеров, какие у них IP. Снаружи виден только роутер. Это как если бы в здании был один общий вход, а внутри — сотни квартир. Входящий не знает, сколько там квартир и какие у них номера.  
  3. Гибкость внутренней сети  
  Ты можешь менять свои внутренние IP как хочешь, переставлять устройства — интернету это безразлично. Снаружи всё равно торчит только один внешний адрес.

#### Типы NAT  
1. **SNAT (Source NAT)** — изменение исходного адреса  
   Когда внутренние устройства выходят в интернет, их IP заменяется на внешний. Самый частый случай.
2. **DNAT (Destination NAT)** — изменение адреса назначения  
   Когда пакет приходит из интернета на внешний IP роутера, его можно перенаправить на конкретное устройство внутри. Это называется проброс портов (port forwarding).  
   **Пример:** ты хочешь, чтобы из интернета можно было зайти на твой домашний сервер. Настраиваешь в роутере: "Все входящие на порт 80 (веб) перенаправляй на 192.168.1.100:80".

#### Таблица NAT в роутере

|Внутренний IP:порт	|Внешний IP:порт	|Куда ходили|
|-|-|-|
|192.168.1.10:54321	|85.23.45.67:12345	|yandex.ru:80|
|192.168.1.20:34567|	85.23.45.67:12346	|google.com:443|
|192.168.1.15:9876|	85.23.45.67:12347	|vk.com:443|

Роутер держит эту таблицу, пока идёт разговор. Когда соединение закрывается — запись удаляется.
  </details>
  
  ### 15. Что такое прокси? Обратный прокси? В чем разница?
  <details>
    <summary>Ответ</summary>
      
  **Прямой прокси (forward proxy)** — это промежуточный сервер между клиентом и интернетом. Клиент (например, твой браузер) отправляет запрос не напрямую на целевой сайт, а на прокси-сервер, а тот уже идёт на сайт от своего имени.

    Ты (192.168.1.10) → Прокси-сервер (45.67.89.10) → Сайт (yandex.ru)
  #### Зачем нужен прямой прокси?
1. Анонимность — сайт видит IP прокси, а не твой реальный.
2. Обход блокировок — если сайт заблокирован в твоей стране, можно пойти через прокси в другой стране.
3. Кэширование — прокси может сохранять копии популярных сайтов и отдавать их быстрее (экономия трафика).
4. Фильтрация — в офисе могут запретить соцсети через прокси.

#### Пример из жизни:  
Ты в офисе, хочешь зайти в Facebook, а он заблокирован. Ты настраиваешь браузер на прокси-сервер в Германии — и всё открывается. Офисный файервол видит только трафик на немецкий прокси, а что ты там смотришь — не видит.

**Обратный прокси** — это промежуточный сервер между интернетом и сервером (или серверами). Клиент думает, что общается напрямую с сервером, но на самом деле запрос сначала попадает на обратный прокси, а тот уже распределяет его дальше.  

    Клиент → Обратный прокси (nginx, 5.255.255.70) → Сервер(ы) (backend)
#### Зачем нужен обратный прокси?
1. Балансировка нагрузки — если у тебя 5 бэкенд-серверов, прокси распределяет запросы между ними.  
2. SSL-терминация — прокси расшифровывает HTTPS, а бэкендам отдаёт обычный HTTP (им легче).
3. Сжатие и кэширование — прокси может сжимать ответы gzip и кэшировать статику (картинки, CSS).
4. Защита — скрывает реальные IP бэкендов, может фильтровать плохие запросы (DDoS).
5. Единая точка входа — даже если у тебя много сервисов (site.com, api.site.com, admin.site.com), они все могут быть за одним прокси.

#### Пример из жизни:  
Ты заходишь на `yandex.ru`. На самом деле запрос приходит на nginx (обратный прокси), который смотрит: "Ага, запрос на `/pogoda` — отправлю на сервер погоды, запрос на `/market` — на сервер маркета". Ты об этом даже не догадываешься.

|Характеристика|	Прямой прокси (Forward proxy)|	Обратный прокси (Reverse proxy)|
|-|-|-|
|Кто за ним?|	Клиенты (пользователи)|	Серверы (бэкенды)|
|Для кого?|	Для клиентов (чтобы спрятаться)	|Для серверов (чтобы спрятать их и распределять нагрузку)|
|Кто знает о нём?|	Клиент знает, что идёт через прокси	|Клиент НЕ знает, он думает, что общается напрямую с сервером|
|Типичная задача	|Обойти блокировки, скрыть IP|	Балансировка, SSL-терминация, кэширование|
|Пример	|Squid, SOCKS-прокси|	Nginx, HAProxy, Apache (в режиме proxy)|

Прямой прокси работает от имени клиентов, скрывая их от серверов. Обратный прокси работает от имени серверов, скрывая их от клиентов и распределяя нагрузку. Прямой нужен пользователям (обход блокировок, анонимность), обратный — владельцам сайтов (балансировка, безопасность, SSL-терминация). Nginx чаще всего используют именно как обратный прокси.
</details>

### 16. Что такое балансировка нагрузки? Какие алгоритмы знаете? (Round Robin, Least Connections и т.д.)?
   <details>
    <summary>Ответ</summary>

 **Балансировка нагрузки** — это распределение входящих запросов между несколькими серверами, чтобы ни один сервер не перегружался, а система в целом работала стабильно и быстро.  
 Простыми словами: если у тебя один сервер — он может не справиться с потоком клиентов. Балансировщик (load balancer) встаёт перед серверами и решает: "Этот запрос отдам первому, этот — второму, этот — третьему".

 #### Зачем нужна балансировка?  
1. Отказоустойчивость — если один сервер упал, балансировщик отправляет запросы на другие.
2. Масштабирование — можно добавить ещё серверов, и система выдержит больше нагрузки.
3. Оптимизация — запросы распределяются так, чтобы серверы не простаивали и не перегревались.

Клиенты видят только балансировщик (один IP). А внутри уже балансировщик решает, кому отдать запрос.

#### Основные алгоритмы балансировки:

1. Round Robin (круговой перебор)  
**Как работает:** Запросы распределяются по кругу: первому, второму, третьему, потом опять первому, второму, третьему...  
**Плюсы:** Простой, равномерное распределение, если серверы одинаковые.  
**Минусы:** Не учитывает нагрузку на сервер. Если один сервер медленнее — он будет забиваться очередями.  
**Пример:** Nginx по умолчанию использует Round Robin.

        Запрос 1 → Сервер A
        Запрос 2 → Сервер B
        Запрос 3 → Сервер C
        Запрос 4 → Сервер A
        Запрос 5 → Сервер B
        ...
2. Least Connections (наименьшее количество соединений)  
**Как работает:** Балансировщик отправляет запрос тому серверу, у которого сейчас меньше всего активных соединений.  
**Плюсы:** Учитывает реальную нагрузку, хорошо работает, если запросы обрабатываются с разной скоростью.  
**Минусы:** Чуть сложнее в реализации, нужно отслеживать количество соединений.  
3. IP Hash (по IP-адресу клиента)  
**Как работает:** Балансировщик вычисляет хеш от IP клиента и по нему определяет сервер. Один и тот же клиент всегда попадает на один и тот же сервер.  
**Плюсы:** Решает проблему "прилипших сессий" (session stickiness) — если сервер хранит состояние клиента.  
**Минусы: Если сервер упадёт** — клиент потеряет данные, и распределение может быть неравномерным.  
4. Weighted Round Robin (взвешенный круговой перебор)  
**Как работает:** Как обычный Round Robin, но у серверов есть вес. Более мощным серверам даём вес 3, слабым — вес 1. На 3 слабых запроса придёт 1 мощный.  
**Плюсы:** Учитывает разную мощность серверов.  
**Минусы:** Веса нужно настраивать руками.  
5. Least Response Time (наименьшее время ответа)  
**Как работает:** Балансировщик отправляет запрос серверу, который быстрее всех отвечал в последнее время.  
**Плюсы:** Хорошо для приложений, где время ответа сильно варьируется.  
**Минусы:** Сложнее в реализации, нужно постоянно замерять время. 
6. Random (случайный выбор)  
**Как работает:** Просто случайный сервер.  
**Плюсы:** Очень просто.  
**Минусы:** Может быть неравномерным статистически.

#### Какой алгоритм выбрать?

|Ситуация|	Алгоритм|
|-|-|
|Все серверы одинаковые, задачи лёгкие	|Round Robin|
|Серверы разной мощности|	Weighted Round Robin|
|Запросы обрабатываются с разной скоростью|	Least Connections|
|Нужна "прилипчивость" (сессии на сервере)|	IP Hash|
|Критично время ответа	|Least Response Time|
  </details>

  ### 1. От кого по умолчанию запускается контейнер? Почему это плохо?
  <details>
    <summary>Ответ</summary>
  По умолчанию Docker использует root (UID 0).
  Ваш контейнер – это еще один процесс, запущенный пользователем root на ядре вашего хоста.
  Это означает, что уязвимость в вашем приложении, среде выполнения Docker или ядре Linux может позволить злоумышленникам выйти из контейнера и выполнить операции с правами root на вашей машине.

  #### Меры безопасности:
  **Запуск контейнеризированных приложений от имени пользователя, не являющегося привилегированным пользователем.**
  Наилучшей практикой для контейнерных приложений является запуск от имени обычного пользователя.
  Большинству программ не нужен root-доступ, поэтому смена пользователя обеспечивает немедленный уровень защиты от взлома контейнера.
  Вы должны создать новую учетную запись пользователя как один из последних этапов в вашем Dockerfile.

  Этого можно добиться с помощью инструкции `USER`:

    FROM base-image:latest
    RUN apt install demo-package
    USER demo-user:demo-group
  
  `USER` часто указывается как предпоследняя ступень в Dockerfile.
  Это означает, что вы все еще можете выполнять операции, требующие root, на более ранних этапах сборки образа.
  Инструкция `apt install` в приведенном выше примере имеет законную потребность в `root`.
  Если бы инструкция `USER` была размещена выше, `apt` был бы запущен от имени `demo-user`, который не имел бы необходимых прав.
  Поскольку инструкции Dockerfile относятся только к сборкам образов, а не к запущенным контейнерам, безопасно оставить изменение пользователя на более поздний период в Dockerfile.

  Изменение пользователя, под которым запускается контейнер, может потребовать обновления разрешений на файлы и папки, к которым он обращается.
  Установите права на все пути, которые будут использоваться вашим приложением:

    COPY initial-config.yaml /data/config.yaml
    USER demo-user:demo-group
    RUN chown demo-user:demo-group /data
  
  или

  Можно воспользоваться сокращенным вариантом, используя флаг `–chown` вместе с `copy`:

    COPY --chown=demo-user:demo-group initial-config.yaml /data/config.yaml

  `--user` флаг при запуске контейнера как альтернатива, описанному выше.
  </details>

### 2. В чем разница между образом и контейнером?

**Контейнер** — это запущенный экземпляр образа, у которого есть свой процесс и свой слой для записи данных. Если удалить контейнер, все записанные в него данные пропадут, а образ останется.

### 3. Отличие контейнерезации от виртуализации?
  <details>
    <summary>Ответ</summary>
    
  **Виртуализация** — это когда мы запускаем несколько полноценных операционных систем на одном физическом сервере, используя гипервизор. Каждая виртуальная машина (ВМ) имеет своё ядро, свои драйверы, свои библиотеки.  
**Контейнеризация** — это когда мы запускаем несколько изолированных процессов на одном ядре операционной системы. Контейнеры используют ядро хоста, но видят только свою часть файловой системы и ресурсов.

|Характеристика	|Виртуализация (VM)	|Контейнеризация (Docker)|
|-|-|-|
|Ядро ОС	|У каждой ВМ своё ядро|	Общее с хостом|
|Запуск	|Минуты (загрузка ОС)|	Секунды (запуск процесса)|
|Размер|	Гигабайты (целая ОС)	|Мегабайты (только приложение)|
|Изоляция|Полная (аппаратный уровень)|	Частичная (на уровне ядра)|
|Ресурсы|	Большие накладные расходы|	Почти как нативные процессы|
|Гостевые ОС	|Можно любые (Windows, Linux)	|Только Linux (или Windows с ограничениями)|

1. Гипервизор эмулирует железо для каждой ВМ. Каждая ВМ думает, что работает на своём компьютере.
2. Контейнеры — это просто обычные процессы Linux, которые изолированы с помощью:  
   - Namespaces — чтобы не видели чужие процессы, сеть, файлы  
   - Cgroups — чтобы не съели все ресурсы
  
### Когда что выбирать?
#### Выбираем виртуализацию, если:  
- Нужно запускать разные ОС (Windows + Linux)
- Требуется максимальная безопасность (аренда серверов разным клиентам)
- Нужно изолировать ядро (например, тестирование драйверов)
- Есть legacy-приложения под старые ОС

#### Выбираем контейнеризацию, если:
- Нужно быстро масштабировать микросервисы
- Важна скорость запуска
- Надо экономить ресурсы (на одном сервере могут жить сотни контейнеров)
- Все приложения под одну ОС (обычно Linux)
</details>

### 4. Что такое Dockerfile? Перечислите основные инструкции?
  <details>
    <summary>Ответ</summary>

**Dockerfile** — это текстовый файл с инструкциями, по которым Docker собирает образ.  
Когда ты выполняешь `docker build`, Docker читает этот файл и выполняет инструкции по порядку, создавая слои образа.

### Основные инструкции:

1. FROM — базовый образ - Указывает, от какого образа отталкиваться.

        FROM ubuntu:22.04
        FROM python:3.9-slim
        FROM alpine:latest
2. RUN — выполнить команду при сборке - Запускает команды внутри образа (установка пакетов, создание папок).

        RUN apt-get update && apt-get install -y nginx
        RUN pip install -r requirements.txt
        RUN mkdir /app
3. COPY — скопировать файлы - Копирует файлы с хоста в образ.

        COPY . /app
        COPY nginx.conf /etc/nginx/nginx.conf
        COPY --chown=user:group file.txt /data/
4. WORKDIR — рабочая директория - Задает директорию, в которой будут выполняться все следующие команды.

        WORKDIR /app
        # Теперь RUN, COPY, CMD будут выполняться в /app
5. CMD — команда по умолчанию - Задает команду, которая выполнится при запуске контейнера. Может быть переопределена при `docker run`

        CMD ["python", "app.py"]
        CMD nginx -g 'daemon off;'
6. ENTRYPOINT — основная команда - Задает команду, которая всегда выполняется при запуске. Её сложнее переопределить.

        ENTRYPOINT ["python", "app.py"]
        CMD ["app.py"]  # app.py можно заменить другим скриптом при запуске
  </details>

### 5. В чем разница между CMD и ENTRYPOINT?
  <details>
    <summary>Ответ</summary>

`CMD` задаёт команду и аргументы по умолчанию, которые можно полностью заменить, просто указав другую команду при `docker run`.  
`ENTRYPOINT` задаёт основную команду, которая всегда выполняется, её нельзя заменить простым указанием команды — можно только добавить аргументы. Но `ENTRYPOINT` тоже можно переопределить, если использовать флаг `--entrypoint`.
  </details>

### 6. Что такое слои в Docker? Как кэширование слоев помогает ускорить сборку?
  <details>
    <summary>Ответ</summary>

**Слой (layer)** — это набор изменений в файловой системе образа.  
Каждая инструкция (кроме тех, что только задают метаданные) в Dockerfile создаёт новый слой, который содержит разницу (дельту) между состоянием до и после выполнения инструкции.  

#### Создают слои:

    FROM ubuntu:22.04           # слой 1: базовый образ
    RUN apt-get update && \      # слой 2: установка пакетов
        apt-get install -y python3
    COPY . /app                  # слой 3: копирование файлов
    RUN pip install -r req.txt   # слой 4: установка зависимостей
#### Не создают слои:

    WORKDIR /app                 # просто запоминает директорию
    CMD ["python", "app.py"]     # команда по умолчанию
    EXPOSE 80                    # открытый порт
    ENV VAR=value                # переменная окружения
    LABEL version="1.0"          # метка
    USER appuser                 # пользователь

Docker кэширует каждый слой после сборки. При повторной сборке он сравнивает инструкцию с кэшем:
- Инструкция не изменилась → Docker берёт готовый слой из кэша (мгновенно)
- Инструкция изменилась → Docker пересоздаёт этот и все последующие слои

      FROM python:3.9-slim        # слой в кэше — берём
      COPY requirements.txt .      # если файл не менялся — берём из кэша
      RUN pip install -r req.txt   # если файл не менялся — берём из кэша
      COPY . .                     # код меняется часто — пересоздаётся
      RUN pytest                   # пересоздаётся
Если бы мы написали COPY . . до RUN pip install, то при каждом изменении кода пришлось бы переустанавливать все зависимости — минуты ожидания.

#### Оптимизация сборки (важно для DevOps!)
Правило порядка: от редко меняющегося к часто меняющемуся

    # 1. Редко меняется
    FROM python:3.9-slim
    WORKDIR /app
    
    # 2. Меняется только при обновлении зависимостей
    COPY requirements.txt .
    RUN pip install --no-cache-dir -r requirements.txt
    
    # 3. Меняется часто (код)
    COPY . .
    
    # 4. Запуск
    CMD ["python", "app.py"]

  #### Как посмотреть слои образа?

    docker history myimage:latest
  </details>

### 7. Как Docker изолирует контейнеры друг от друга и от хоста? (про namespaces и cgroups)
  <details>
    <summary>Ответ</summary>

Docker изолирует контейнеры с помощью двух механизмов ядра Linux: namespaces и cgroups. Namespaces изолируют "точку зрения" процесса — у каждого контейнера свои PID, сеть, файловая система, имя хоста. Cgroups ограничивают потребление ресурсов — CPU, RAM, диск, чтобы один контейнер не мог "съесть" весь сервер. Вместе они создают лёгкую, но надёжную изоляцию, при которой контейнер — это просто процесс на хосте, который думает, что он отдельный сервер.

#### Как это работает вместе (на примере)  
Когда ты запускаешь docker run nginx:
1. Создаются пространства имён для нового контейнера:
   - Новый PID namespace (процесс nginx будет иметь PID 1 внутри)
   - Новый NET namespace (свой виртуальный Ethernet)
   -  Новый MNT namespace (своя корневая ФС)
   -  так далее...

2. Создаются cgroups:
   - Создаётся папка `/sys/fs/cgroup/memory/docker/<container-id>`
   - Туда записываются ограничения (если заданы)
   - PID nginx добавляется в эти cgroups

3. Процесс запускается, но:
   - Он "думает", что работает на отдельном сервере (namespaces)
   - Ядро следит, чтобы он не съел больше разрешённого (cgroups)
  
#### Команды для исследования

    # Посмотреть cgroups контейнера
    docker inspect <container-id> | grep -i pid
    ls -l /sys/fs/cgroup/memory/docker/
    
    # Посмотреть namespaces процесса на хосте
    ls -l /proc/<pid>/ns/
  </details>

### 9. Какие есть best practices для написания Dockerfile?
  <details>
    <summary>Ответ</summary>

1. Запускать только один процесс на контейнер.
2. Стараться объединять несколько команд RUN в одну для уменьшения количества слоёв образа.
3. Частоизменяемые слои образа необходимо располагать ниже по уровню, чтобы ускорить процесс сборки, т.к. при изменении верхнего слоя, все нижеследующие слои будут пересобираться.
4. Указывать явные версии образов в инструкции FROM, чтобы избежать случая, когда выйдет новая версия образа с тегом latest.
5. При установке пакетов указывать версии пакетов.
6. Очищать кеш пакетного менеджера и удалять ненужные файлы после выполненной инструкции.
7. Использовать multistage build для сборки артифакта в одном контейнере и размещении его в другом.
8. Для безопасности использовать инструкцию `USER`
  </details>

### 10. Как пробросить порт контейнера на хост?
  <details>
    <summary>Ответ</summary>

Проброс портов (port forwarding) нужен, чтобы сервис внутри контейнера стал доступен снаружи — с хоста, другим контейнерам или из внешней сети.

    docker run -p 8080:80 nginx
- 8080 — порт на хосте (твоей машине)
- 80 — порт в контейнере (где слушает nginx)

После этой команды ты можешь открыть в браузере http://localhost:8080 и попадёшь в nginx внутри контейнера.

Проброс порта делается флагом -p при docker run или в docker-compose через секцию ports. Формат: -p <порт на хосте>:<порт в контейнере>. Можно указать протокол (TCP/UDP) и привязать к конкретному IP на хосте. Без проброса порт контейнера доступен только внутри сети Docker, но не с хоста. Под капотом Docker добавляет правила в iptables, которые перенаправляют трафик с порта хоста на IP контейнера.
  </details>

### 11. Какие типы сетевых драйверов используются в docker??
  <details>
    <summary>Ответ</summary>

Docker поддерживает несколько сетевых драйверов: bridge (по умолчанию, для изоляции на одном хосте), host (контейнер использует сеть хоста без изоляции), none (без сети), container (общий стек с другим контейнером), overlay (для кластеров Swarm) и macvlan (контейнер получает MAC-адрес в физической сети). Выбор зависит от задачи: bridge — для большинства приложений, overlay — для распределённых систем, host — для производительности.

1. **bridge (мост)** — используется по умолчанию  
    Создаёт виртуальный сетевой мост (docker0) на хосте. Контейнеры подключаются к этому мосту и получают IP из подсети 172.17.0.0/16. Они могут общаться друг с другом по IP, но снаружи (с хоста) доступны только через проброшенные порты.
    
        docker run --network bridge nginx  # можно не писать, это по умолчанию
2. host (хост)
    Контейнер использует сетевой стек хоста напрямую. У него нет своего IP — он слушает на IP хоста. Проброс портов (-p) не нужен и не работает.

       docker run --network host nginx
       # Теперь nginx доступен на порту 80 хоста сразу
3. none (нет сети)
    У контейнера нет сетевых интерфейсов (только loopback). Полная изоляция.

       docker run --network none alpine sleep 1000
4. container (общий стек с другим контейнером)
    Контейнер использует сетевой namespace другого контейнера. У них общий IP, общие порты.

        docker run --network container:nginx-container alpine
        # Этот alpine будет иметь тот же IP, что и nginx-container
5. overlay (наложенная сеть)
    Создаёт сеть, которая работает поверх сетей хостов и позволяет контейнерам на разных физических машинах общаться так, будто они в одной локальной сети.

        docker network create -d overlay my-cluster-net
6. macvlan
    Каждому контейнеру назначается реальный MAC-адрес, и он выглядит в физической сети как отдельное устройство (получает IP от DHCP сети).

        docker network create -d macvlan \
        --subnet=192.168.1.0/24 \
        --gateway=192.168.1.1 \
        -o parent=eth0 my-macvlan-net

#### Как посмотреть и создать сети
        
        # Список сетей
        docker network ls
        
        # Создать свою bridge-сеть
        docker network create --driver bridge --subnet 10.10.0.0/16 my-net
        
        # Запустить контейнер в этой сети
        docker run --network my-net --name app1 nginx
        
        # Подключить второй контейнер к той же сети
        docker run --network my-net --name app2 alpine ping app1
        # app2 сможет пинговать app1 по имени!
  </details>  
</details>
